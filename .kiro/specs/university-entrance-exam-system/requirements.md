# Requirements Document

## Introduction

The University Entrance Examination System is a web-based application for RV University (RVU) that manages the complete entrance examination workflow. The system enables students to apply for admission by selecting schools and programs, allows administrators to approve or reject applications, and provides teachers with access to view students assigned to their respective schools. The system enforces role-based access control across three user types: Admin, Teacher, and Student.

## Glossary

- **System**: The University Entrance Examination System web application
- **Student**: A user who registers to apply for admission to one or more schools at RVU
- **Teacher**: A user assigned to exactly one school who can view students who selected that school
- **Admin**: A user with full system control who approves or rejects student applications
- **School**: An academic division within RVU (e.g., School of Computer Science & Engineering, School of Business)
- **Program**: A degree or course offering within a school (e.g., B.Tech CSE, BBA)
- **Application_Status**: The approval state of a student account (pending, approved, rejected)
- **Selected_Schools**: The collection of schools and programs a student chooses during registration
- **Assigned_School**: The single school that a teacher is responsible for
- **Dashboard**: The role-specific interface displayed after successful authentication
- **Username**: A unique identifier auto-generated for each user in the format: firstname + 3-digit random number
- **Auto_Generated_Password**: A secure random password generated by the system containing 8-10 characters with uppercase, lowercase, and numbers
- **First_Login_Flag**: A boolean indicator (isFirstLogin) that tracks whether a user must change their password

## Requirements

### Requirement 1: Student Registration with Auto-Generated Credentials

**User Story:** As a prospective student, I want to register for the entrance examination system by providing my details and selecting schools and programs, so that I can apply for admission to RVU with secure auto-generated login credentials.

#### Acceptance Criteria

1. THE System SHALL provide a registration page at /signup accessible without authentication
2. WHEN a student submits the registration form, THE System SHALL collect name, email, password, and phone number
3. WHEN a student submits the registration form, THE System SHALL allow selection of multiple schools from the available RVU schools
4. WHEN a student submits the registration form, THE System SHALL allow selection of programs based on the selected schools
5. WHEN a student completes registration, THE System SHALL automatically generate a unique Username in the format: firstname + 3-digit random number (e.g., apoorva123, rahul456)
6. WHEN a Username collision occurs, THE System SHALL regenerate the 3-digit number until a unique Username is found
7. WHEN a student completes registration, THE System SHALL automatically generate a secure random Auto_Generated_Password containing 8-10 characters with uppercase letters, lowercase letters, and numbers
8. WHEN a student completes registration, THE System SHALL hash the Auto_Generated_Password using bcrypt before storing it in the database
9. WHEN a student completes registration, THE System SHALL store the Username, hashed password, role as STUDENT, status as ACTIVE, and First_Login_Flag as TRUE
10. WHEN a student completes registration, THE System SHALL display the generated credentials on screen with the message: "Your login credentials: Username: [username], Password: [password]"
11. WHEN a student completes registration, THE System SHALL store the selected schools and programs as structured data containing school name and program name pairs
12. WHEN a student completes registration, THE System SHALL set the Application_Status to "pending"

### Requirement 2: Authentication with Username and Password

**User Story:** As a user, I want to log in to the system using my username and password, so that I can access my role-specific dashboard securely.

#### Acceptance Criteria

1. THE System SHALL provide a login page at /login accessible without authentication
2. THE System SHALL require Username and password for authentication (NOT email)
3. WHEN a user submits valid Username and password, THE System SHALL authenticate the user by comparing the hashed password stored in the database
4. WHEN a Student with Application_Status "pending" logs in successfully, THE System SHALL display the message "Waiting for admin approval" and prevent dashboard access
5. WHEN a Student with Application_Status "approved" and First_Login_Flag TRUE logs in, THE System SHALL redirect to the password change page at /change-password
6. WHEN a Student with Application_Status "approved" and First_Login_Flag FALSE logs in, THE System SHALL redirect to /student/dashboard
7. WHEN a Teacher logs in, THE System SHALL redirect to /teacher/dashboard
8. WHEN an Admin logs in, THE System SHALL redirect to /admin/dashboard
9. WHEN a user submits invalid Username or password, THE System SHALL display the error message "Invalid username or password"
10. THE System SHALL ensure the Admin user exists in the database with Username "admin", securely hashed password, role ADMIN, and status ACTIVE

### Requirement 2A: Mandatory First Login Password Change

**User Story:** As a new user, I want to be required to change my auto-generated password on first login, so that I have a secure password that only I know.

#### Acceptance Criteria

1. THE System SHALL provide a password change page at /change-password accessible only to authenticated users with First_Login_Flag TRUE
2. WHEN a user with First_Login_Flag TRUE accesses /change-password, THE System SHALL display input fields for "New Password" and "Confirm Password"
3. WHEN a user submits the password change form, THE System SHALL validate that "New Password" and "Confirm Password" match
4. WHEN a user submits the password change form with matching passwords, THE System SHALL hash the new password using bcrypt
5. WHEN a user submits the password change form with matching passwords, THE System SHALL update the user's password in the database
6. WHEN a user submits the password change form with matching passwords, THE System SHALL set First_Login_Flag to FALSE
7. WHEN a user successfully changes their password, THE System SHALL redirect to their role-specific dashboard
8. WHEN a user with First_Login_Flag FALSE attempts to access /change-password, THE System SHALL redirect to their dashboard

### Requirement 3: Admin Approval System

**User Story:** As an admin, I want to review and approve or reject student applications, so that I can control who gains access to the examination system.

#### Acceptance Criteria

1. THE System SHALL provide an admin dashboard at /admin/dashboard accessible only to Admin users
2. WHEN an Admin accesses the dashboard, THE System SHALL display all student registrations with Application_Status "pending"
3. WHEN an Admin views a student record, THE System SHALL display name, email, Selected_Schools, selected programs, and Application_Status
4. WHEN an Admin approves a student, THE System SHALL update the Application_Status to "approved"
5. WHEN an Admin rejects a student, THE System SHALL update the Application_Status to "rejected"
6. WHEN a Student's Application_Status changes to "approved", THE System SHALL grant login access to the student dashboard

### Requirement 4: Teacher School Assignment and Student Viewing

**User Story:** As a teacher, I want to view only the students who selected my assigned school, so that I can focus on applicants relevant to my school.

#### Acceptance Criteria

1. THE System SHALL assign exactly one Assigned_School to each Teacher user
2. THE System SHALL provide a teacher dashboard at /teacher/dashboard accessible only to Teacher users
3. WHEN a Teacher accesses the dashboard, THE System SHALL display only students whose Selected_Schools includes the Teacher's Assigned_School
4. WHEN a Teacher views a student record, THE System SHALL display the student's name, email, and the program selected for the Teacher's Assigned_School
5. THE System SHALL exclude students who did not select the Teacher's Assigned_School from the teacher dashboard view

### Requirement 5: Student Dashboard Access

**User Story:** As an approved student, I want to access my dashboard to view my selected schools and available exams, so that I can proceed with the entrance examination process.

#### Acceptance Criteria

1. THE System SHALL provide a student dashboard at /student/dashboard accessible only to Student users with Application_Status "approved"
2. WHEN an approved Student accesses the dashboard, THE System SHALL display the list of schools the student selected during registration
3. WHEN an approved Student accesses the dashboard, THE System SHALL display available exams for each selected school
4. WHEN a Student with Application_Status "pending" or "rejected" attempts to access /student/dashboard, THE System SHALL deny access and redirect to the login page

### Requirement 6: Role-Based Access Control

**User Story:** As a system administrator, I want to enforce role-based access control, so that users can only access pages and data appropriate to their role.

#### Acceptance Criteria

1. WHEN a non-authenticated user attempts to access /admin/dashboard, THE System SHALL redirect to /login
2. WHEN a non-authenticated user attempts to access /teacher/dashboard, THE System SHALL redirect to /login
3. WHEN a non-authenticated user attempts to access /student/dashboard, THE System SHALL redirect to /login
4. WHEN a Student attempts to access /admin/dashboard or /teacher/dashboard, THE System SHALL deny access and display an authorization error
5. WHEN a Teacher attempts to access /admin/dashboard or /student/dashboard, THE System SHALL deny access and display an authorization error
6. WHEN an Admin attempts to access /teacher/dashboard or /student/dashboard, THE System SHALL deny access and display an authorization error

### Requirement 7: School and Program Data Management

**User Story:** As a system administrator, I want the system to maintain accurate school and program data, so that students can select from valid options during registration.

#### Acceptance Criteria

1. THE System SHALL store data for approximately 9 schools representing RVU's academic divisions
2. THE System SHALL store multiple programs for each school
3. WHEN a student accesses the registration form, THE System SHALL display all available schools for selection
4. WHEN a student selects one or more schools, THE System SHALL display programs associated with the selected schools
5. THE System SHALL prevent selection of programs that do not belong to the selected schools

### Requirement 8: User Data Persistence

**User Story:** As a developer, I want user data to be persisted in the database using Prisma, so that the system maintains data integrity and supports efficient queries.

#### Acceptance Criteria

1. THE System SHALL store user records with fields: id, name, email, username, password, role, status, phone, selectedSchools, assignedSchool, isFirstLogin, and createdAt
2. WHEN a user record is created, THE System SHALL enforce that the username field is unique across all users
3. WHEN a user record is created, THE System SHALL store the username in the format: firstname + 3-digit random number
4. WHEN a user record has role "student", THE System SHALL store selectedSchools as JSON containing school and program pairs
5. WHEN a user record has role "teacher", THE System SHALL store assignedSchool as a string containing the school name
6. WHEN a user record has role "admin", THE System SHALL not require selectedSchools or assignedSchool fields
7. THE System SHALL hash passwords using bcrypt before storing them in the database
8. THE System SHALL enforce unique email addresses across all user records
9. THE System SHALL store isFirstLogin as a boolean field with default value TRUE for new users
10. THE System SHALL use email for contact purposes only and NOT for authentication

### Requirement 9: Teacher Assignment by Admin

**User Story:** As an admin, I want to create teacher accounts and assign them to schools, so that each school has exactly 2 teachers responsible for entrance exams.

#### Acceptance Criteria

1. THE System SHALL allow Admin users to create Teacher accounts without requiring teacher self-registration
2. WHEN an Admin creates a Teacher account, THE System SHALL require assignment of exactly one Assigned_School
3. THE System SHALL allow each school to have exactly 2 assigned teachers
4. WHEN an Admin creates a Teacher account, THE System SHALL set the role to "teacher" and Application_Status to "approved"
5. THE System SHALL provide the created Teacher with login credentials to access /teacher/dashboard

### Requirement 10: Application Data Integrity

**User Story:** As a system administrator, I want the system to maintain data integrity for student applications, so that school and program selections remain consistent throughout the application lifecycle.

#### Acceptance Criteria

1. WHEN a Student's application is stored, THE System SHALL preserve the exact schools and programs selected during registration
2. WHEN a Teacher views a student record, THE System SHALL display only the program the student selected for the Teacher's Assigned_School
3. WHEN an Admin views a student record, THE System SHALL display all Selected_Schools and programs the student chose
4. THE System SHALL prevent modification of Selected_Schools after initial registration
5. FOR ALL student records, the selectedSchools JSON structure SHALL contain valid school and program pairs that existed at registration time

### Requirement 11: Admin User Management View

**User Story:** As an admin, I want to view all users in the system with their usernames and roles, so that I can manage user accounts and reset passwords when needed.

#### Acceptance Criteria

1. THE System SHALL provide an admin user management page at /admin/users accessible only to Admin users
2. WHEN an Admin accesses the user management page, THE System SHALL display a list of all users with Name, Username, and Role
3. WHEN an Admin views the user list, THE System SHALL exclude password information from the display
4. WHERE password reset functionality is implemented, THE System SHALL allow Admin users to reset a user's password
5. WHEN an Admin resets a user's password, THE System SHALL generate a new secure random password
6. WHEN an Admin resets a user's password, THE System SHALL set the user's First_Login_Flag to TRUE
7. WHEN an Admin resets a user's password, THE System SHALL display the new password to the Admin for communication to the user

### Requirement 12: API Endpoints for Authentication System

**User Story:** As a developer, I want well-defined API endpoints for the authentication system, so that the frontend can interact with the backend securely and consistently.

#### Acceptance Criteria

1. THE System SHALL provide a POST /api/register endpoint that accepts name, email, phone, and selectedSchools
2. WHEN POST /api/register is called, THE System SHALL return the auto-generated Username and Auto_Generated_Password in the response
3. THE System SHALL provide a POST /api/login endpoint that accepts Username and password (NOT email)
4. WHEN POST /api/login is called with valid credentials, THE System SHALL return a session token and user information including role, status, and First_Login_Flag
5. THE System SHALL provide a POST /api/change-password endpoint that accepts newPassword and confirmPassword
6. WHEN POST /api/change-password is called by an authenticated user, THE System SHALL validate password match, update the hashed password, and set First_Login_Flag to FALSE
7. WHERE admin user management is implemented, THE System SHALL provide a GET /api/users endpoint accessible only to Admin users
8. WHERE password reset is implemented, THE System SHALL provide a POST /api/admin/reset-password endpoint that accepts userId and returns the new generated password

### Requirement 13: Password and Username Validation

**User Story:** As a system administrator, I want the system to enforce strong password and username requirements, so that user accounts remain secure.

#### Acceptance Criteria

1. WHEN the System generates a Username, THE System SHALL ensure it is unique by checking against all existing usernames in the database
2. WHEN a Username collision is detected, THE System SHALL regenerate the 3-digit random number and retry until a unique Username is found
3. WHEN the System generates an Auto_Generated_Password, THE System SHALL ensure it contains at least one uppercase letter, one lowercase letter, and one number
4. WHEN the System generates an Auto_Generated_Password, THE System SHALL ensure the length is between 8 and 10 characters
5. WHEN a user changes their password on first login, THE System SHALL validate that the new password is at least 8 characters long
6. WHEN a user submits a password change, THE System SHALL validate that "New Password" and "Confirm Password" fields match exactly
7. THE System SHALL display appropriate error messages when password validation fails
8. THE System SHALL never store passwords in plain text format
